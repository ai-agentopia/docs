# Deterministic Delivery Start Front Door

> Status: CLOSED/DEFERRED — C2 rejected, separate ingress deferred pending gateway fork
> Epic: P0.5
> Upstream of: P1 (Execution Authorization Enforcement)
> Last updated: 2026-03-22

---

## 1. Problem Statement

There is an upstream gap before workflow start that P1 does not cover. When a user sends a delivery request to the orchestrator bot, the system currently depends on LLM prompt compliance to call `start_delivery`. If the LLM does not call the tool — replying with freeform prose instead — no workflow starts, but the user may see text implying pipeline activity.

This is a **delivery-start-path problem**, not an execution-authorization problem.

---

## 2. Settled Diagnosis

- Upstream gap is real and separate from P1
- Freeform orchestrator chat is soft before workflow/policy engages
- `start_delivery` is the canonical typed internal contract (unchanged)
- LLM may not call `start_delivery` — prompt/SOUL guidance is not enforcement
- System cannot currently distinguish "LLM chose not to call tool" from "LLM failed to call tool"
- For a user delivery request, the system must end in exactly one of:
  1. `workflow_started` (with evidence: workflow_id, temporal_workflow_id)
  2. `clarification_required` (structured, not freeform)
  3. `delivery_not_started` (explicit, not silent)

---

## 3. Safe-Now Scope (Proven Capabilities Only)

These items are buildable with current proven capabilities. No unknown dependencies.

| # | Item | What it delivers | Dependency |
|---|---|---|---|
| S1 | Delivery-start idempotency | `delivery_start_requests` table. Same `delivery_request_id` → same workflow. Universal prerequisite for any delivery-start path | None |
| S2 | Delivery-start observability | Counters: attempted, succeeded, replayed, failed, not_started | None |

S1 is required regardless of which delivery-start path is chosen (hook, fork, separate ingress, or current tool-based). It is not wasted work.

## 4. Unknown Boundaries

The following capabilities are NOT proven in the current gateway/plugin SDK:

| Unknown | Why it matters | Current evidence |
|---|---|---|
| `before_agent_start` can safely perform mutating POST | Needed for hook-based `/deliver` (Option C2) | Hook proven for context injection only. Mutating side effects unverified |
| `event.prompt` contains full literal user message | Needed to detect and parse `/deliver` command in hook | `event.prompt` used by relay/wf-bridge, but exact content for Telegram commands unverified |
| Stable replay-safe message identity in hook context | Needed for idempotency dedup if hook retries | Unknown. Hook may only expose prompt text |
| `chat_id` / session identity in hook context | Needed for multi-turn clarification | Unknown. Only `event.prompt` proven |
| Bot functions normally with `start_delivery` removed from tool surface | Needed for Option C2 (prevent LLM double-start) | Untested |

**Hard rule:** No solution path that depends on an unknown boundary may be chosen before spike verification.

## 5. Spike Plan

All spikes run on a **dedicated probe bot only**. No production rollout. No user-facing changes.

### Spike A — Hook Side-Effect Safety

- **Question:** Can `before_agent_start` safely perform a mutating HTTP POST?
- **Method:** Test hook in wf-bridge that calls a dummy POST endpoint. Send 20 messages. Count invocations per message. Measure failure behavior.
- **Pass:** Hook fires cleanly per message. Hook failure does not crash gateway.
- **Fail:** Multiple fires per message, or hook failure crashes/hangs gateway.
- **Timebox:** 1 day

### Spike B — Prompt Visibility

- **Question:** Does `event.prompt` contain the full literal `/deliver ...` text including arguments?
- **Method:** Log `event.prompt` in hook. Send various `/deliver` commands on Telegram.
- **Pass:** Full text visible, no truncation, no modification.
- **Fail:** Text preprocessed, prefix stripped, or arguments truncated.
- **Timebox:** Half day

### Spike C — Session Identity (Informational)

- **Question:** Does hook context expose `chat_id` or equivalent session identity?
- **Method:** Log all event fields.
- **Pass:** Session identity accessible.
- **Fail:** Only prompt text available.
- **Timebox:** Half day
- **Note:** Not blocking for live gate. Informational for multi-turn clarification scope.

### Spike D — Tool Surface Removal

- **Question:** Does bot function normally with `start_delivery` removed from tool surface?
- **Method:** Remove `start_delivery` from trial bot configmap. Chat with bot about non-delivery topics.
- **Pass:** Bot works, no errors.
- **Fail:** Gateway errors or degraded behavior.
- **Timebox:** Half day

### Spike E — Stable Request Identity

- **Question:** Does hook/ingress expose a retry-stable upstream message identity?
- **Method:** Inspect hook event payload for `message_id`, `update_id`, or equivalent Telegram-level stable identifier.
- **Pass:** Same message replayed → same identity value.
- **Fail:** No replay-stable identity source. Hook-generated UUID is not dedup-safe on retry.
- **Timebox:** Half day
- **Note:** **Mandatory for live gate.** "Observed once" in Spike A ≠ replay safety guarantee.

## 6. Decision Gate

After spikes complete, choose ONE path:

| A | B | D | E | Decision |
|---|---|---|---|---|
| PASS | PASS | PASS | PASS | **OPTION_C2_LIVE_VIABLE.** Hook-based `/deliver` with idempotency. No fork needed |
| PASS | PASS | PASS | FAIL | **OPTION_C2_REJECTED.** Hook works but no replay identity → cannot dedup retries. Separate ingress or fork needed |
| PASS | FAIL | any | any | **SEPARATE_INGRESS_OR_FORK_REQUIRED.** Hook cannot parse `/deliver` from event.prompt |
| FAIL | any | any | any | **GATEWAY_FORK_REQUIRED.** Hook not safe for mutating side effects |

**Allowed gate outcomes:**
- `OPTION_C2_LIVE_VIABLE` — hook path proven safe. Build Option C2
- `OPTION_C2_REJECTED` — hook works but no replay safety. Choose: separate ingress OR gateway fork
- `SEPARATE_INGRESS_OR_FORK_REQUIRED` — hook cannot parse. Choose: separate ingress OR fork
- `GATEWAY_FORK_REQUIRED` — hook fundamentally unsafe. Fork is the only path

**Hard rule:** Do NOT recommend live Option C2 unless A + B + D + E all pass.
**Hard rule:** E FAIL does NOT automatically mean fork required — separate ingress may also solve it.

## 9. Deferred Workstream — Separate Ingress

**Status:** DEFERRED pending gateway fork (#25). Permanent dual-lane model adopted instead.

### Decision Record (2026-03-22)
Separate ingress investigation concluded:
- Spike F research: gateway owns Telegram connection via OpenClaw framework (opaque binary). No interception point exists before gateway.
- Spike G research: bot-config-api can sendMessage but cannot receive updates without conflicting with gateway.
- Cannot add second Telegram consumer (polling or webhook) for same bot token.
- **Conclusion:** separate ingress requires gateway fork — deferred to milestone #25.

### Permanent Dual-Lane Model
Instead of deterministic start-path, the system adopts soft dual-lane:
- **Delivery lane:** `start_delivery` tool, guided by SOUL contract
- **Consultation lane:** relay tools, always open for orchestrator
- **Known gap:** start-path integrity relies on LLM prompt compliance, not code enforcement
- **Accepted trade-off:** until gateway fork enables Telegram adapter intercept

### Previous Direction (archived)
Dedicated service/endpoint handles `/deliver` command directly, bypassing gateway LLM entirely. Bot-config-api or a lightweight companion service owns the `/deliver` inbound path.
