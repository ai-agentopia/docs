## 2. Purpose

Add Trello as a one-way planning projection for workflow orchestration.

Authoritative state remains:
- workflow engine / Temporal
- GitHub milestone / issue / PR state

Trello is added as:
- a stakeholder-facing planning board
- a read-only projection of workflow/GitHub state
- a visualization surface exposed in the product UI

This milestone does **not** make Trello authoritative.

## 3. Final Product Contract for M6.2

### In scope
1. Create exactly one Trello card per workflow.
2. Move the card based on workflow state transitions.
3. Refresh card metadata from GitHub events without moving the card.
4. Persist Trello linkage and sync state in Postgres.
5. Surface Trello status in workflow detail UI.
6. Keep Trello failures non-blocking for workflow execution.
7. Future-proof the service boundary, persistence shape, and action model for later sync work.

### Out of scope
1. Trello -> GitHub/workflow sync.
2. Trello webhooks.
3. GitHub reverse-sync webhooks.
4. Conflict resolution and loop prevention engine.
5. Manual resync endpoints.
6. MCP-based repair/admin tooling.
7. Trello-driven workflow transitions.
8. Trello board auto-creation.

Deferred scope is tracked in issue #276.

## 4. Architecture Decisions

### 4.1 Service boundary
M6.2 uses a provider-neutral service boundary:
- `PlanningBoardService` (abstract)
- `TrelloBoardAdapter` (concrete, Trello REST API)
- `NullBoardAdapter` (no-op when disabled)

Workflow code does not call Trello directly.

### 4.2 Runtime path
Workflow/Trello integration is triggered through a separate Temporal activity:
- `sync_planning_board`

This activity is fire-and-forget from the workflow perspective:
- failures are caught
- sync errors are persisted
- parent workflow continues unconditionally

### 4.3 Startup and bootstrap model
When `TRELLO_ENABLED=true`:
1. app startup runs a best-effort Trello preflight
2. adapter resolves board metadata from `TRELLO_BOARD_ID`
3. adapter fetches board lists
4. adapter auto-creates missing canonical lists if needed
5. real sync paths still perform lazy/idempotent ensure as needed

This keeps Trello visible early without making it a hard startup dependency.

## 5. Minimal Infra / Secret Contract

Infra provides only:
- `TRELLO_ENABLED`
- `TRELLO_API_KEY`
- `TRELLO_API_TOKEN`
- `TRELLO_BOARD_ID`

Infra does **not** require manual configuration for:
- list IDs
- board name
- card URL templates

Vault path: `secret/data/agentopia/trello`
Loaded via multi-path init container: `VAULT_SECRETS_PATHS=secret/data/agentopia/gateway,secret/data/agentopia/trello`

## 6. Canonical Trello Lists

Phase-1 canonical lists (auto-created if missing):
1. `Planned`
2. `In Progress`
3. `In Review`
4. `QA Approved`
5. `Merge Ready`
6. `Done`
7. `Blocked`

## 7. State and Event Mapping

### 7.1 Workflow state -> Trello list movement
| Workflow state | Trello target | Runtime wired |
|---|---|---|
| `PLANNED` | `Planned` | Yes |
| `IN_DEV` | `In Progress` | Yes |
| `DEV_DONE` | `In Review` | Yes |
| `QA_CHANGES_REQUESTED` | `In Review` | Yes |
| `QA_APPROVED` | `QA Approved` | Yes |
| `MERGE_READY` | `Merge Ready` | Yes |
| `DONE` | `Done` | Yes |
| `BLOCKED` | `Blocked` | Yes — via `workflow_service._sync_trello_for_state()` |
| `CANCELED` | archive card | Yes |

Runtime coverage: **9 of 9** workflow states. BLOCKED/unblock goes through `workflow_service.py` command path and is wired via `_sync_trello_for_state()` helper. Issue #277 closed.

### 7.2 GitHub events -> metadata refresh only
These events update card content only and must not move the card:
- PR opened (populates pr_number, branch from head ref)
- PR synchronized (refreshes head SHA)
- review submitted (notes review state)
- PR merged (notes merge status)
- issue closed (notes issue status)

There is no separate `branch_created` phase-1 event.

## 9. Sync Action Contract

`OutboundSyncAction` dataclass with typed fields:
- `action`: create_card | move_card | refresh_metadata | archive_card
- `source_system`: workflow | github
- `source_event_type`
- `workflow_state`
- `idempotency_key` (auto-generated, deterministic)
- `metadata`

### Idempotency key strategy
- `workflow:{id}:create`
- `workflow:{id}:state:{state}`
- `workflow:{id}:github:{event_type}:{identifier}`
- `workflow:{id}:archive`

Note: `OutboundSyncAction` exists and is tested but is not yet the canonical runtime payload through the activity. The activity still uses `SyncPlanningBoardInput` directly. Wiring through `OutboundSyncAction` is an M6.3 prerequisite.

## 10. Card Content Rule

Trello card content is projection only.

Allowed: render workflow ID, repo, milestone/issue/branch/PR, acceptance criteria, objective, status text.

Not allowed: parsing authoritative linkage back from Trello description.

Normalized linkage remains in DB/service persistence.

## 11. UI Contract

Workflow detail UI shows:
- Trello card link
- board name
- current list/status
- sync status badge (synced/error/pending)
- last synced timestamp
- last error if any
- **"Read-only projection"** label

API response: `GET /workflows/{id}` returns `trello` object with `sync_mode: "projection"` or `null` if no mapping.
